# 外部工具借鉴

本文件记录从公开 AI prompt / agent workflow 工具中吸收的设计启发。它不是核心协议，不能覆盖 `核心协议/` 中的当前规则。

## Fabric

项目：<https://github.com/danielmiessler/fabric>

可借鉴点：

- 把 AI 能力按真实任务拆成可复用 pattern。
- pattern 应该能被不同工具调用，而不是绑死在某个聊天窗口。
- 好的提示词库需要可发现、可组织、可版本化。

已吸收为：

- `08-模式库与流程评测.md` 的“模式卡片”。
- README 中“核心协议不写死平台工具名称”的公开共享原则。

## PromptFlow

项目：<https://microsoft.github.io/promptflow/>

可借鉴点：

- LLM 应用不只是 prompt，还包括流程、工具、调试、评测和部署。
- 流程需要 tracing、evaluation、CI/CD 这类质量反馈机制。

已吸收为：

- `08-模式库与流程评测.md` 的流程状态、评测记录和回归检查。

## promptfoo

项目：<https://github.com/promptfoo/promptfoo>

可借鉴点：

- prompt、agent、RAG 都应该能被测试。
- 模型选择和提示词质量应尽量用用例和指标比较，而不是只凭感觉。
- 对高风险 AI 应用要有安全和红队意识。

已吸收为：

- 最小评测集、标准评测、完整评测。
- 失败用例和不合格表现字段。

## OpenAI Evals

项目：<https://github.com/openai/evals>

可借鉴点：

- 评测是理解模型版本变化和系统变化影响的重要方式。
- 私有工作流也可以建立自己的小型 eval，不必公开敏感数据。

已吸收为：

- `协议验收用例.md` 和 `08-模式库与流程评测.md` 的回归用例思路。

## DSPy

项目：<https://dspy.ai/>

可借鉴点：

- 把任务写成输入输出签名，比堆长提示词更可维护。
- 模块、签名、评测指标可以让 AI 流程更容易迭代。

已吸收为：

- 模式卡片中的“输入签名”和“输出签名”。
- 禁止假设、必需输出、不允许输出等字段。

## LangGraph

项目：<https://github.com/langchain-ai/langgraph>

可借鉴点：

- 长流程 agent 需要状态、检查点、人工介入和可恢复机制。
- 复杂流程要能观察执行路径，而不是只看最终回答。

已吸收为：

- 流程状态、人工确认点、长流程交接字段。

## AutoGen / CrewAI

项目：

- <https://github.com/microsoft/autogen>
- <https://github.com/crewAIInc/crewAI>

可借鉴点：

- 多 Agent 协作需要明确角色、任务、交接和人工 review。
- 高自主流程和显式控制流程应区分，不能所有任务都交给自动 agent 自由发挥。

已吸收为：

- 多 Agent 场景中的人工确认点。
- “低模型做搬运，高模型做关键判断”的边界。
- 模式卡片中的角色、输入输出和失败模式要求。

## 对本工具箱的定位修正

这些工具大多偏 prompt 库、LLM 应用开发、agent 编排或评测框架。本工具箱不直接复制它们的工程实现，而吸收它们的协议层原则：

- 从 Fabric 吸收“模式卡片化”。
- 从 PromptFlow 吸收“流程可调试、可评测”。
- 从 DSPy 吸收“输入输出签名”。
- 从 promptfoo / Evals 吸收“用例评测和回归检查”。
- 从 LangGraph / CrewAI / AutoGen 吸收“状态、人工介入和多 Agent 交接”。
