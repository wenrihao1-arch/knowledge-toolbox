# 需求解析 v1.1（Requirement Analysis）

**定位：** 工具箱的唯一路由中枢。每次对话静默运行，不归档。深读/深析/深研/搜索收集不再各自判断是否触发——统一由需求解析路由。
**在工具箱中的位置：** 前置层。所有用户输入先过需求解析，再路由到目标工具。

---

## 一、核心原则

1. **用户显式指令优先。** 用户说了"深析""深读""深研""搜一下"→ 直接路由，跳过自动判定。用户不需要知道内部架构——他们只是说了想做什么。
2. **不明确才问，明确就静默。** 大多数问题能自动判定。只有确实有歧义且答偏风险高时才追问。
3. **单一真相源。** 深读/深析/深研内部的"触发规则"表已替换为"适用范围"——它们不再各自做路由决策。路由只看需求解析。
4. **路由决策传递。** 需求解析的判定结果作为上下文传递给下游工具，避免下游重新解读。

---

## 二、路由判定

### 第0步（最优先）：用户显式指令

| 用户说 | 路由 |
|--------|------|
| "深读一下" / "精读这篇" | → 深读 |
| "深析一下" / "分析一下这个事件" | → 深析 |
| "深研一下" / "研究一下" / "系统了解" | → 深研 |
| "搜一下" / "帮我查一下" | → 搜索收集 |

> **一旦匹配显式指令，跳过后续所有自动判定。**

### 第1步：识别类型（无显式指令时）

| 类型 | 识别特征 | 示例 |
|------|---------|------|
| **文章** | 发链接、粘贴长文、"看看这篇" | "https://... 帮我看看" |
| **事件** | 问新闻、问XX怎么回事、有时效性+涉及多方 | "Apple刚发的这个政策怎么回事" |
| **领域** | 开放式的系统了解、"帮我理一下XX" | "研究一下储能行业" |
| **具体查询** | 事实型问题、对比、数据、有明确答案 | "目前有哪些公司在做..." |
| **操作请求** | 要求执行操作而非分析 | "帮我改一下这个文件" |
| **模糊/混合** | 范围不清、同时涉及多个类型 | "AI最近有什么进展" |

### 第2步：判定路径

| 类型 | 路由 |
|------|------|
| 文章 | → **深读** |
| 事件 | → **深析** |
| 领域 | → **深研** |
| 具体查询（需搜索/查证） | → **搜索收集** |
| 具体查询（已知，不需搜索） | → **直接回答** |
| 操作请求 | → **直接执行** |
| 模糊/混合 | → **追问澄清** |

**边界模糊判定（深度搜索 vs 浅度深研）：**
这是最容易模糊的边界。判断标准：问题的核心是"回答一个具体问题"还是"系统了解一个领域"？

| 场景 | 判定 | 理由 |
|------|------|------|
| "AI编程助手谁做的最好？" | 搜索收集（深度） | 核心是回答"谁最好"，有明确终点 |
| "研究一下AI编程助手赛道" | 深研 | 核心是"系统了解"，没有明确终点 |
| "了解一下"介于两者之间 | 默认走搜索收集 | 更轻更快，不够可以随时升级 |

**默认倾向：** 边界模糊时默认走搜索收集——它更轻更快，且深度搜索的结果如果不够，可以随时升级到深研。

### 第3步：识别缺口

追问条件（同时满足）：问题确实模糊有歧义 + 不问会导致答偏 + 不是"范围稍宽但仍可给出合理回答"

追问格式：一句话，不给选项列表。例如："你是想了解市场格局，还是想选一个产品用？"

### 第4步：降级/升级建议

路由不是"判了就结束了"。如果目标工具的筛选门控可能拦截，提前考虑替代路径。

| 场景 | 处理 |
|------|------|
| 有可能信息不足的事件 | 先路由到深析，但预备如果深析筛选不通过，降级为搜索收集 |
| 范围可能太大的领域 | 先追问缩小范围，或先路由到搜索收集探路 |
| 质量存疑的文章 | 正常路由到深读，深读评分会处理 |

---

## 三、路由传递

需求解析的判定结果传递给下游工具：

```
来源: 需求解析
判定: 深析-争议冲突版
原因: 事件+多方纠纷
预备: 如信息不足→降级搜索收集
```

下游工具收到后：
- 跳过自己的触发判定（路由已定）
- 仍执行自己的内部门控（评分的评分、筛选的筛选）
- 门控不通过时，按预备降级路径走，不直接终止

---

## 四、路由示例

| 用户说 | 解析 | 路由 |
|--------|------|------|
| "深析一下Apple这个政策" | 显式指令→深析 | 深析 |
| "帮我看看这篇" + 链接 | 文章 | 深读 |
| "现在有哪些AI编程助手？" | 具体查询+需搜索 | 搜索收集→回答 |
| "Rust和Go哪个适合做后端" | 已知知识 | 直接回答 |
| "AI最近有什么进展" | 模糊+范围宽 | 追问方向 |
| "研究一下储能行业" | 显式指令→深研 | 深研 |

---

## 五、与子工具的关系

```
需求解析（唯一路由中枢）
  │
  ├──→ 搜索收集（信息收集，不归档）
  │      └── 结果供深读/深析/深研/直接回答消费
  │
  ├──→ 深读（文章精读，评分筛选仍是内部门控）
  ├──→ 深析（事件分析，速览筛选仍是内部门控）
  ├──→ 深研（领域研究，价值评估仍是内部门控）
  │
  └──→ 直接回答/执行
```

**关键：** 需求解析负责"往哪走"，目标工具负责"走多深"。目标工具的筛选门控若拦截，走降级路径而非终止。
