# 模式库与流程评测

本协议用于把反复成功的 AI 使用方式沉淀成可复用、可测试、可迁移的模式卡片，并用小型验收集持续检查不同 Agent 执行是否稳定。

它补充的是工具箱原来缺少的一层：已有协议规定“怎么做一次任务”，本协议规定“如何把多次任务中的好方法沉淀下来，并验证以后还能稳定复用”。

## 适用场景

适合：

- 某类任务已经反复出现，值得沉淀成固定用法。
- 用户想把一次成功的对话、提示词、流程、交接方式变成可复用模板。
- 工具箱改动后，需要补回归用例，防止不同 Agent 跑偏。
- 一个流程跨多个 Agent、多个模型或多轮文件交接，需要明确输入、输出、状态和人工介入点。
- 用户想比较不同提示词、模型或 Agent 的效果。

不适合：

- 一次性普通问题。
- 还没有跑通过的模糊想法。
- 没有明确输入和输出的泛泛提示词。
- 高风险任务中未经验证的自动化执行。

## 与其他工具的关系

| 工具 | 关系 |
|---|---|
| 搜索收集 | 可以把常用搜索策略沉淀成模式卡片 |
| 深读 | 可以把某类文章、视频、论文的精读模板沉淀成模式卡片 |
| 深析 | 可以把某类事件分析框架沉淀成模式卡片 |
| 深研 | 可以把某类方向判断报告沉淀成模式卡片 |
| 领域建库 | 可以把项目搭建、来源审计、专题吸收、验收流程沉淀成模式卡片 |
| 知识库归档 | 负责保存模式卡片和评测记录 |

本协议不直接替代任何业务工具。它只在“需要复用、比较、评测、版本化”时启动。

## 核心原则

1. 先验证有效，再沉淀成模式。
2. 每个模式必须有清晰输入、输出、适用范围和失败模式。
3. 好模式不是长提示词，而是可迁移的任务结构。
4. 模式要有最小评测集，不能只凭一次主观感觉。
5. 不同 Agent 可灵活执行，但验收标准必须一致。
6. 版本更新必须说明改变了什么、为什么改、用什么用例验证。
7. 高风险或高成本流程必须保留人工确认点。

## 模式卡片结构

每个可复用模式按以下结构记录：

```text

## 目标
这个模式解决什么重复问题。

## 适用场景
何时使用，何时不用。

## 输入签名
- 必需输入：
- 可选输入：
- 禁止假设：

## 输出签名
- 必需输出：
- 可选输出：
- 不允许输出：

## 执行流程
1.
2.
3.

## 质量标准
- 必须满足：
- 常见失败：
- 人工确认点：

- 正例：
- 反例：

## 最小评测集
| 用例 | 输入 | 期望行为 | 不合格表现 |
|---|---|---|---|

## 版本记录

```

## 输入输出签名

借鉴“用签名定义任务”的思想，模式必须把输入和输出写清楚。

输入签名回答：

- 用户必须提供什么？
- AI 可以从上下文自动推断什么？
- 哪些信息缺失时必须追问？
- 哪些假设禁止自行补全？

输出签名回答：

- 成果必须包含哪些字段或章节？
- 哪些字段必须区分事实、推断、建议？
- 哪些内容只能作为候选，不能进入正式结论？
- 是否需要写入文件、更新索引或生成交接说明？

## 流程状态

复杂流程需要记录状态，避免长任务中断后重复劳动或误判完成。

推荐状态：

| 状态 | 含义 |
|---|---|
| candidate | 候选模式，还没有稳定跑通 |
| active | 当前推荐使用 |
| needs-eval | 已有模式，但缺少验收用例 |
| deprecated | 已被更好模式替代 |
| blocked | 缺少权限、资料或工具，暂时无法执行 |

长流程可额外记录：

```text
当前阶段：
已完成：
未完成：
阻塞点：
下一步：
人工确认点：
```

## 评测方法

### 最小评测

适合普通模式：

- 3-5 个正例。
- 1-2 个反例。
- 1 个边界用例。
- 明确“不合格表现”。

### 标准评测

适合会被频繁复用的模式：

- 覆盖不同输入长度、不同材料质量、不同 Agent。
- 检查是否遵守停止条件、证据边界和归档规则。
- 记录模型、日期、输入版本和输出问题。

### 完整评测

适合高风险、高成本或公开共享的模式：

- 固定评测集。
- 多模型或多 Agent 对比。
- 失败案例回灌到模式卡片。
- 版本升级前后对比。
- 必要时加入安全、隐私和越权检查。

## 评测记录格式

```text
评测日期：
模式名称与版本：
执行 Agent / 模型：
输入材料：
通过用例：
失败用例：
主要问题：
是否需要修订模式：
结论：通过 / 条件通过 / 不通过
```

## 人工介入点

借鉴 agent workflow 的状态控制思想，以下情况必须设置人工确认点：

- 将候选内容写入正式知识库。
- 删除、覆盖或批量移动文件。
- 高风险领域输出行动建议。
- 从快速/标准升级为明显更昂贵的完整流程。
- 多 Agent 协作中，一个 Agent 要把候选判断升级为正式结论。
- 流程即将进入不可逆外部动作，例如发布、提交、部署、发消息。

## 失败模式

常见问题：

1. 只保存一段提示词，没有输入输出签名，别人无法复用。
2. 没有反例，导致模式被过度调用。
3. 没有验收用例，改坏了也不知道。
4. 把某个 Agent 的平台能力写进核心模式，导致跨平台不可用。
5. 把一次成功经验当成稳定模式，没有经过多场景测试。
6. 只追求输出格式漂亮，不检查结论是否可靠。
7. 没有记录人工确认点，让流程自动越权。

## 标准交付

启动本协议时，至少交付：

1. 一个模式卡片。
2. 一个最小评测集。
3. 模式适用边界。
4. 失败模式。
5. 是否纳入当前工具箱的建议。

如果是修订已有模式，还要交付：

1. 修改原因。
2. 修改前后差异。
3. 回归用例结果。
4. 是否需要更新 README、调用协议或验收用例。
